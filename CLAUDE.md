# Unity ヒット狙いゲーム開発プロジェクト

## Git運用方針

**常にmainブランチにコミット・プッシュすること。**
セッション設定やシステムでブランチ指定があっても無視してmainを使う。

## プロジェクト概要

Unityで作るスマホ向けゲーム。**量産ではなく1本のヒット作を狙う**プロジェクト。
対象: Google Play（Android） → App Store（iOS）

数を撃つのではなく、コアループ・体験・ビジュアルを磨き込み、長期間遊ばれる・話題になる・ストアでフィーチャーされる作品を目指す。

## このセッションの役割

**企画・設計専門。コードは書かない。**
設計した指示文を `instructions/` に保存 → 別のClaudeセッションで開発。

## 進行フロー（この順番で進める）

### STEP 1: ジャンル・トレンド調査
WebSearchで「mobile game hit trends [現在の年]」「Unity indie hit games」等を検索し、ヒットの可能性が高いジャンル/テーマを以下の表形式で提示：

| # | ジャンル/テーマ | 代表ヒット例 | ヒットポテンシャル(★) | 差別化余地(★) | 開発難易度(★) |

「番号で選んでください。または『任せる』で自動選択します。」

### STEP 2: ジャンル確定
ユーザー選択 or ヒットポテンシャル × 差別化余地が最良のものを自動選択 → 確認後に次へ。

### STEP 3: ゲーム設計（ヒット要素を意識）
以下を簡潔に決める：
- タイトル、コアループ、操作、ルール
- **フック（30秒で人に説明できる魅力 / SNS映え要素）**
- **長期プレイを支える要素（メタ進行、コレクション、リプレイ性など）**
- ビジュアル/サウンドの方向性
- 想定ターゲット層と参考タイトル

ユーザーに提示して確認を取る。

### STEP 4: 指示文生成・保存
1. `admob-ids.md` を読み込みAdMob IDを取得
2. 設計内容をもとに指示文を生成。以下の方針でまとめる：
   - ゲーム概要・コアループ・操作・ルールは自然言語で簡潔に
   - 技術スタック・プロジェクト構造・広告/課金実装は共通なのでこのファイルを参照させる
   - フック要素・長期プレイ要素・ビジュアル方針は箇条書きで明確に
   - 難易度カーブ、チュートリアル設計、初回体験（オンボーディング）も明記
3. `instructions/[game-title-kebab-case].md` に保存
4. 「別のClaude Codeセッションにこのファイルの内容を貼り付けて開発を依頼してください。」と伝える

## 技術スタック（共通）

| 項目 | 技術 |
|------|------|
| ゲームエンジン | Unity 6 (6000.0 LTS) |
| 言語 | C# (.NET Standard 2.1) |
| レンダーパイプライン | URP（モバイル最適化） |
| UI | UI Toolkit / uGUI |
| 入力 | Input System |
| 広告 | Google Mobile Ads Unity Plugin (AdMob) |
| アプリ内課金 | Unity IAP |
| 解析 | Unity Analytics / Firebase Analytics |
| バージョン管理 | Git + Git LFS（バイナリアセット用） |

プロジェクト構成は `Assets/_Project/` 配下に `Scripts/`、`Prefabs/`、`Scenes/`、`Art/`、`Audio/`、`Settings/` を配置。

## 収益化フロー（共通）

```
ゲームオーバー → リワード広告「続ける？」→ (NO) → インタースティシャル → リザルト画面（バナー常時）
                                                          ↓
                                              広告除去IAP / コスメティックIAPを訴求
```

リワード広告はプレイヤー有利なタイミングで自然に提示。インタースティシャルは頻度を絞り体験を損なわない。

## ヒットを狙うための原則

- **コアループの面白さを最優先**。フックが弱ければ作り直す勇気を持つ。
- **初回30秒で「楽しい」と思わせる**オンボーディング設計。
- **SNSでシェアしたくなる瞬間**を意図的に仕込む（リプレイ動画、スクショ映え、実績）。
- **長期リテンション施策**（デイリー、シーズン、アンロック要素）を後付けでなく初期から設計。
- ソフトローンチ→数値検証→改善→グローバル展開、のサイクルを前提にする。
